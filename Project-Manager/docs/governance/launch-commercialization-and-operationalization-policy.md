# Launch Commercialization and Operationalization Policy

## Purpose

Define mandatory launch-proximal planning artifacts so projects do not reach launch without commercialization, marketing, and SOP readiness.

This policy applies near launch, not at day-zero onboarding.

---

## Trigger Window

A project enters launch-proximal governance when any of the following are true:
- launch date is committed
- pilot window is scheduled with external users/customers
- commercialization motion is approved
- customer-facing delivery is expected within the next planning cycle

---

## Required Launch-Proximal Artifacts

- commercialization-plan.md
- marketing-plan.md
- operationalization-sops.md

These may be generated from templates and expanded per project type.

---

## Operationalization Minimums

Operationalization SOP coverage must include:
- intake-to-delivery operating flow
- escalation and incident handling flow
- release and rollback procedure
- handoff and coverage procedure
- reporting cadence and owner responsibilities

---

## Path 2 White-Label Brand Identity

For projects using Path 2 commercialization motion:
- command center admin must support a white-label brand identity mode
- active brand identity must be visible in admin command center metadata
- brand identity selection must be documented before launch approval

---

## Enforcement

A launch-proximal project is NOT launch-ready until:
- commercialization plan exists and is approved
- marketing plan exists and is approved
- operationalization SOP set exists and is approved
- Path 2 brand identity mode is configured when applicable
